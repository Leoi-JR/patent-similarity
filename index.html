Core Algorithm

算法核心：结构信号 × 语义信号的加权融合

单纯依赖关键词无法理解语义；单纯依赖向量可能跨领域误判；单纯依赖分类粒度太粗。 把"技术结构"与"语义内容"两种信号以不同权重叠加，是这套系统最核心的设计判断。

similarity = IPC结构相似度 × α
+ 摘要向量相似度 × β
+ 标题向量相似度 × γ

IPC 结构信号确立技术领域边界，防止跨领域误判；语义向量信号深入技术内容层面， 捕捉字面以外的语义接近；两者互补，比任一单独维度都更稳定可靠。 最终结果裁剪到 [0, 1]， 阈值过滤 + top-k 截断后写入 Parquet。

α

基于 IPC 分类树的稀疏独热重叠计算，确立技术领域边界。 防止不同领域因措辞接近而被误判为相似专利。

β

摘要包含更完整的技术方案描述，是最强的语义判断依据， 与 IPC 信号同权重，共同主导最终分数。

γ
标题向量相似度（Title）

标题通常更短、更泛化，语义信息密度低于摘要， 因此只作为辅助排序信号，而非主要判定基础。

IPC Structural Similarity

IPC 结构相似度：稀疏独热矩阵 × 层级权重

越靠近叶节点，说明技术分类越细、越接近，权重也越高。 这种表达既保留了分类树的层级信息，又能借助稀疏矩阵乘法做大规模高效计算。

IPC 分类树示例 · 层级权重分配
G 物理 └ G06 计算；推算；计数 └ G06F 电数字数据处理 × w₃ └ G06F 17 专用计算功能 × w₄ └ G06F 17/16 矩阵运算 × w₅
IPC_sim = L3 × w₃ + L4 × w₄ + L5 × w₅

两个专利的 IPC 代码在某层节点相同，该层得 1，否则为 0。 三层加权求和即为结构相似度，最大值为 1（所有层完全一致）。

数据结构

稀疏独热矩阵（CSR）

对 level3 / level4 / level5 分别建稀疏矩阵。 每行对应一个专利，每列对应一个唯一 IPC 节点， 命中则置 1，其余为 0。专利数量大时内存效率显著优于稠密表示。

scipy.sparse.csr_matrix → cupy cp_csr_matrix
GPU 加速

稀疏矩阵批量点积

CPU 稀疏矩阵转为 CuPy 稀疏矩阵后，直接在 GPU 上执行 batch_feat.dot(feat_matrix.T)， 三层结果按权重累加，与 embedding 稠密点积结果合并。

cp_csr_matrix.dot() · .multiply(weight)
设计权衡

为什么不用全 IPC 路径匹配？

全路径完全匹配要求两个专利拥有同一叶节点，召回率极低； 仅比较顶层分类则粒度太粗、精度不够。 分层加权是在两者之间取得的平衡点。

Semantic Embedding

语义相似度：分字段向量化 + 归一化点积

title 和 brief 分别独立编码为向量文件，在相似度阶段再按权重合成， 而不是直接拼接文本——这样可以在最终分数中精确控制每个字段的贡献比例。

Embedding 模型

中文大模型

iic/nlp_gte_sentence-embedding_chinese-large
输出 1024 维归一化向量，专为中文语义匹配设计。 通过 ModelScope 本地缓存加载，无需外部 API。

编码策略

分字段独立编码

title 与 brief 分别生成独立的 .npz 文件， 内含 embeddings 和 ids 两个数组。 相似度阶段再按字段权重加权合成，保持精确控制。

服务形式

HTTP 服务 per GPU

embedding_server.py 封装为 Flask HTTP 服务， 每个 GPU 启动一个实例，提供 /embed 接口。 天然支持多卡横向并行，无状态设计便于扩展。

相似度计算

点积 = 余弦相似度

模型输出的向量已经过 L2 归一化，因此两向量的点积 等价于余弦相似度，无需额外除以模长。 适合在 GPU 上做批量矩阵乘法 cp.dot(A, B.T)。

Pipeline Architecture

工程架构：三步解耦的批处理流水线

每一步只解决一个明确问题：先稳定生产向量，再批量计算相似度，最后把结果写成可消费的结构化文件。步骤之间通过文件解耦，互不依赖，便于单独重跑。

1

启动向量服务

按 GPU 数量启动多个 embedding_server.py 实例，每个实例绑定一个 GPU，对外暴露 /embed HTTP 接口。服务无状态，天然支持横向扩展。

embedding_server.py
2

批量生成向量

扫描 patent_data/ 下所有 IPC 文件，用 ThreadPoolExecutor 并行分发到各服务。每个 IPC 生成 title / brief 两套 .npz 文件。

generate_embedding.py
3

计算相似度

读取 embedding 与 IPC 层级数据，用 multiprocessing.Queue 实现多 GPU 自动调度。阈值过滤 + top-k 截断后写入 .parquet。

compute_similarity.py
步骤 2 · 向量化并行策略

ThreadPoolExecutor 多服务分发

Worker 数与 embedding 服务实例数一致。一个 IPC 处理完成后，空闲 worker 立即认领下一个 IPC。任一 IPC 失败记入 failures 列表，主流程结束时统一汇总报错。

步骤 3 · 相似度计算并行策略

multiprocessing.Queue 多 GPU 调度

所有待处理 IPC 入队，每个 GPU 对应一个独立进程，持续 get_nowait() 取任务直到队列为空。GPU 间完全独立，不共享 CuPy 上下文，避免竞争。

中间文件

.npz 是两阶段的桥梁

每个 IPC × 每个字段生成一个 npz，包含 embeddings 和 ids，两阶段通过文件完全解耦，可单独重跑任意步骤。

一致性保证

先校验 id 顺序，再计算

相似度阶段严格校验 title / brief embedding 与专利数据的 id 列表是否顺序一致，不一致立即报错，防止静默错位。

输出约束

不保留全部配对

移除自身匹配，分数裁剪到 [0,1]，阈值过滤 + top-k 截断，确保输出质量可控、体积合理。

Engineering Quality

工程质量：让流水线可以长期稳定运行

算法正确只是起点。一个能在生产中持续运行的系统，还需要可靠的错误处理、数据一致性保障、可观测的测试体系。

并行调度

多 GPU 自动领任务

IPC 任务入队后，空闲 GPU 进程持续 get_nowait() 认领，无需手动分配。任意数量 GPU 均可直接扩展，整体吞吐随 GPU 数线性提升。

数据一致性

先校验，再计算

相似度计算前对比 patent_df['id']、brief_ids、title_ids 三者顺序。任一不一致立即 raise ValueError，杜绝 embedding 错位导致的静默错误结果。

错误策略

快速失败，不静默跳过

单批 embedding 请求失败立即 raise_for_status() 抛出，不降级为空向量。多 IPC 并行时失败记入列表，主流程结束统一 raise RuntimeError，确保错误可见。

内存管理

分批落盘，避免积压

embedding 每攒 save_batch_size 批次写一次 npz；相似度结果每 SIMILARITY_SAVE_BATCH 批次写一次 parquet。大规模数据下不会因内存积压导致 OOM。

规模控制

IPC 分桶 + top-k 截断

按 IPC 分组将 O(n²) 全量比较降为组内比较，从根本上控制计算规模。top-k 截断确保每个专利的近邻数有上界，输出文件体积可预测。

测试体系

Smoke + Integration

Smoke tests：CPU 环境，monkeypatch 模拟 CuPy，验证核心相似度计算主流程。
Integration tests：启动真实 embedding 服务，端到端覆盖向量化 → 相似度计算全链路。
